MindCoach — Onboarding Funnel Tracking

Ready-to-implement event taxonomy for the value-first first-run journey: splash → onboarding → profile → coach match → demo session → premium offer → trial → auth. User is anonymous through every step and only gets identified at the final auth screen. Keep names and property keys exactly as written.

RULES

Implementation rules

Keep these consistent so the funnel reports cleanly in PostHog.

Naming
  • Event names: snake_case, past tense (e.g. onboarding_completed).
  • Property keys: snake_case, no abbreviations.
  • Enum values: snake_case, exactly as listed in this doc.
Identity (anonymous-until-end flow)
  • Stages 1–8 fire while the user is anonymous on the PostHog auto-generated distinct ID. Do NOT call identify() earlier — it'll fragment the funnel.
  • Set user properties (gender, support_area, coach_approach, available_days, time_slot, trial_status, etc.) on the anonymous user via posthog.people.set() as soon as they're known. PostHog will carry them onto the identified user at auth.
  • On auth_completed: call posthog.alias(user_id) first, then posthog.identify(user_id). The alias call stitches the entire anonymous funnel (stages 1–8) into the new user record.
  • For guest users: still call identify() with a stable guest ID (e.g. guest_<uuid>) so they're trackable across sessions.
When to fire view vs. tap
  • View events fire when a screen / element becomes visible — once per session per screen.
  • Tap events fire on the user action (button tap, swipe).
  • Auto events fire automatically (lifecycle / completion).
Funnel definition in PostHog

Configure the main onboarding funnel with these steps in order:

  1. app_opened
  2. onboarding_completed
  3. profile_completed
  4. coach_matches_viewed
  5. video_call_started
  6. demo_session_completed
  7. continue_free_tapped or premium_purchase_tapped
  8. trial_activated
  9. auth_completed (conversion goal)

Build a second funnel for revenue: premium_offer_viewed → premium_purchase_tapped.

PII
  • Never send name or email as an event property. Set them only as user properties.
  • Don't send raw auth tokens, device IDs, or IP — PostHog already handles device/session info.
